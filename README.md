# 📄 프로젝트 개요 (Overview)

## ✦ 프로젝트명

> **이력서 기반 맞춤형 AI 역량 검증 및 압박 면접 플랫폼**

<br>

## ✦ 프로젝트 기간

> 2026.02.27(금) ~ 2026.03.03(화) **(5일)**

<br>

## ✦ 프로젝트 소개

사용자의 이력서(PDF)를 분석하여 직무와 경험에 기반한 맞춤형 면접을 제공하는
초개인화 AI 면접 솔루션입니다.

단순한 1문 1답식 챗봇을 넘어, 지원자의 답변을 실시간으로 분석하고
모순점이나 깊이 있는 기술적 지식을 파고드는 평가 루브릭 기반 압박 꼬리 질문 시스템을 구현했습니다.

또한 WebRTC(웹캠 미러링) 및 STT(음성 인식)를 통해
실제 화상 면접과 동일한 긴장감과 UI/UX를 제공하며, 면접 전 이력서를 다각도로 분석해주는 AI 딥섹션 대시보드를 통해 구직자의 면접 대비 효율을 극대화합니다.

<br>

## ✦ 프로젝트 필요성 (배경)

### ✶ 정형화된 면접 대비의 한계

* 시중 면접 서비스는 뻔한 질문 반복에 그침
* 실제 지원자의 역량 검증에 한계 존재

### ✶ 맞춤형 모의 면접의 부재

* 프로젝트 경험 및 기술 스택 기반 심층 질문 환경 필요

### ✶ 효율적인 채용 검증 도구 필요

* 대량 이력서 검토 및 1차 스크리닝에 시간/비용 과다 소모

<br><br>

# 💼 비즈니스 이해 (Business Understanding)

## ✶ B2C (구직자 대상 에듀테크 SaaS)
* 이력서 기반 무한 반복 AI 모의 면접 제공

* **상위 % 상대평가 대시보드(백분위)** 제공
* 문항별 피드백 및 게이미피케이션 기반 성장 지표 제공

## ✶ B2B (기업용 HR 채용 솔루션)
* 면접관 주관적 편향 최소화 및 객관적 지표(Rubric) 기반 채점
* DB 설계, 알고리즘, CS 지식 등 역량 수치화
* 1차 서류/면접 스크리닝 리소스 절감 및 AI 기반 핵심 역량 자동 추출

# 📄 프로젝트 목표

### ✶ 동적 RAG (Continuous Learning RAG) 구축

* 이력서 데이터 + 실시간 답변을 Vector DB(ChromaDB)에 즉시 임베딩
* 이전 답변 맥락을 기억하고 이력서의 팩트를 기반으로 검증하는 AI 면접관 구현

### ✶ AI 제어 로직 (Hallucination Control)
* 엄격한 JSON Schema와 4가지 평가 루브릭을 적용한 프롬프트 엔지니어링
* 점수(40점 이하)에 따른 동적 꼬리질문 생성(최대 2회) 및 주제 전환 제어

### ✶ Dual DB 아키텍처 설계

* **RDBMS (SQLite)** : 질문 및 메타데이터 관리
* **Vector DB (ChromaDB)** : AI 검색 및 문맥 유지
* 투트랙 구조 분리 및 동기화

<br>

# ✨ 주요 기능 (Key Features)

1. **📄 이력서 기반 맞춤형 질문 (RAG)**
  - 지원자의 PDF를 분석하여 프로젝트 경력에 특화된 기술 질문 추출

2. **🎙️ 실시간 음성 면접 (STT/TTS)**
  - Whisper(STT)와 OpenAI TTS를 활용한 실전 대화 인터페이스

3. **⏱️ 답변 소요 시간 측정**
  - 각 문항별 답변 시간을 측정하여 지원자의 순발력 및 태도 분석

4. **📊 Dual DB 아키텍처**
  - 500개 기술 질문 풀(MySQL)과 개인화 문맥(ChromaDB)의 하이브리드 검색

# ℹ️ Insight

### 💡 기술적 성과

* **프롬프트 엔지니어링 고도화**: 단일 LLM 호출로 [점수, 피드백, 꼬리질문 필요여부, 다음 메인질문 한국어 번역]을 한 번에 뽑아내는 강력한 `JSON Schema` 프롬프트를 설계하여 API 호출 비용과 Latency를 획기적으로 줄였습니다.
* **초개인화 질문 생성**: 단순 랜덤 추출이 아닌, AI가 이력서에서 추출한 기술 스택 키워드를 MySQL의 `question_pool`과 매칭하는 하이브리드 검색을 구현했습니다.
* **비용 최적화 (Caching)**: 한 번 분석된 이력서의 JSON 결과값을 MySQL에 영구 저장(`user_resumes`)하여, 사용자가 보관함에서 이력서를 열람할 때 AI 토큰을 전혀 소모하지 않는 상용 서비스급 최적화를 달성했습니다.

# 🚀 Trouble Shooting (주요 문제 해결 및 아키텍처 개선)

## 1. Infrastructure & Network (인프라 및 네트워크)

### 1.1 브라우저 마이크 권한 차단 및 동적 도메인 문제 (Ngrok 도입)
  * **문제 상황:** 실시간 음성 면접(STT) 기능을 구현했으나, 브라우저 보안 정책상 `http://` 환경에서는 마이크 접근 권한이 차단됨. 이를 해결하기 위해 ngrok(무료)을 사용했으나, 서버 재구동 시마다 도메인이 변경되어 프론트엔드-백엔드 간 API 엔드포인트 단절 현상 발생.
  * **해결 방안:** ngrok 유료 플랜을 도입하여 **고정된 HTTPS Custom Domain**을 확보.
  * **개선 결과:** 브라우저의 WebRTC/마이크 권한을 안정적으로 획득함과 동시에, 서버 재시작 시마다 발생하던 프론트엔드 환경변수 수정 공수를 완전히 제거하여 개발 생산성을 극대화함.

### 1.2 소셜 로그인 Redirect URI 충돌 및 동적 IP 문제 (AWS Elastic IP)

  - **문제 상황:** AWS EC2 인스턴스 재부팅 시 퍼블릭 IP가 동적으로 변경됨. 이로 인해 카카오/구글 소셜 로그인(OAuth 2.0) 설정에 등록해둔 `Redirect URI`가 매번 무효화되어 인증 에러(401)가 발생하는 치명적 문제 발생.
  * **해결 방안:** 유지 비용이 발생하더라도 시스템 안정성이 최우선이라 판단하여, EC2 인스턴스에 **AWS 탄력적 IP(Elastic IP)**를 할당하여 아키텍처의 엔드포인트를 고정함.
  * **개선 결과:** 소셜 로그인 파이프라인이 완벽히 안정화되었으며, 외부 API 연동 시 발생하는 IP 화이트리스트 관리 이슈를 원천 차단함.

## 2. AI & LLM Pipeline (인공지능 파이프라인)

### 2.1 LLM 할루시네이션(환각) 및 JSON 파싱 에러
  * **문제 상황:** AI 면접관이 이력서에 없는 기술을 지어내어 꼬리질문을 하거나, 응답 텍스트에 마크다운과 설명이 섞여 프론트엔드에서 JSON 파싱 에러(`json.decoder.JSONDecodeError`)가 빈번히 발생함.
  * **해결 방안:** 프롬프트 내에 `[🚨 할루시네이션 방지 특별 원칙]`을 도입하여 오직 RAG 검색 결과와 DB의 모범 답안 안에서만 근거를 찾도록 강제함. 또한 OpenAI API의 `response_format={ "type": "json_object" }`를 적용.₩
  * **개선 결과:** 100% 안정적인 JSON 데이터 파싱 구조를 확립하였으며, AI가 자의적으로 내용을 창작하는 현상을 0%로 통제함.

## 3. 인프라 및 데이터베이스 아키텍처 (Infra & DB Architecture)

### 3.1 브라우저 음성 권한(HTTPS) 차단 및 동적 도메인 문제 (Ngrok 도입)
  * **문제 상황:** 실시간 음성 면접(STT) 기능을 구현했으나, 브라우저 보안 정책상 `http://` 환경에서는 마이크 접근 권한(WebRTC)이 차단되어 기능 테스트가 불가함. 이를 임시 해결하기 위해 ngrok(무료)을 썼으나, 서버 구동 시마다 도메인이 바뀌어 API 엔드포인트가 단절됨.
  * **해결 방안:** ngrok 유료 플랜을 도입하여 **고정된 HTTPS Custom Domain**을 확보함.
  * **개선 결과:** 보안 정책을 충족하여 브라우저 마이크 권한을 안정적으로 획득하였으며, 매번 환경변수를 수정해야 했던 비효율을 완벽히 제거함.

### 3.2 소셜 로그인 Redirect URI 충돌 및 동적 IP 문제 (AWS Elastic IP)
  * **문제 상황:** AWS EC2 인스턴스는 재부팅 시 퍼블릭 IP가 동적으로 변경됨. 이로 인해 카카오/구글 소셜 로그인(OAuth 2.0) 설정에 등록해둔 `Redirect URI`가 매번 무효화되어 인증 에러(401)가 발생하는 치명적 문제 발생.
  * **해결 방안:** 소요 비용이 발생하더라도 시스템 안정성이 최우선이라 판단하여, EC2 인스턴스에 **AWS 탄력적 IP(Elastic IP)**를 할당해 서버의 퍼블릭 엔드포인트를 영구적으로 고정함.
  * **개선 결과:** 소셜 로그인 파이프라인 및 콜백(Callback) 처리가 완벽히 안정화되었으며, 외부 API 연동 시 IP 변경으로 인한 장애를 원천 차단함.

### 3.3 로컬 분산 DB로 인한 데이터 파편화 및 통합 테스트 불가
  * **문제 상황:** 프로젝트 초기, 각 팀원이 개별적으로 **로컬 MySQL** 환경을 구축하여 사용함에 따라, 팀원 간 회원 및 면접 데이터가 불일치하고 릴리즈 전 통합 테스트(Integration Test)가 불가능한 상황 발생.
  * **해결 방안:** AWS EC2 인스턴스에 공용 MySQL 서버를 구축하고, 모든 팀원이 하나의 **원격 공유 DB**를 바라보도록 `.env` 설정 및 아키텍처를 전면 전환함.
  * **개선 결과:** 실시간 데이터 동기화 및 DBeaver를 통한 원격 GUI 관리가 가능해졌으며, 무결성 있는 팀 단위 통합 테스트 환경을 성공적으로 구축함.
